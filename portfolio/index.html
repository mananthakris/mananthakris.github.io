01
Multi-Agent Systems Arize Phoenix
BRD → Engineering System Plan Generator
Given any Business Requirements Document, produce a decision-ready engineering plan with competing architectural options — grounded in the company's actual stack, constraints, and team capabilities.
LangGraph Multi-Agent LLM-as-Judge Offline + Online Evals HITL
Details +

Architecture & Key Decisions

  • LangGraph hub-and-spoke with typed GraphState Six specialized agents wired through a single Pydantic state object — making the data contract between agents explicit and testable. The Orchestrator never pre-fetches context; each agent issues its own retrieval query tuned to its specific task.
  • Critic drives a revision loop, not a re-run The Critic scores all upstream outputs on five dimensions. If revision_required and revision_count < MAX_CYCLES, the graph loops back with revision notes — preserving state rather than re-running the full pipeline. That distinction matters for cost and latency at scale.
  • Problem classification gates the graph Solution Architect classifies each BRD as greenfield / new_feature / migration / integration / poc / enhancement before any design work. This single field controls downstream agent behavior — PoC Planner only runs when warranted.
  • Versioned prompt registry All agent system prompts live in a registry with semantic version tags. Every eval run is tagged with prompt versions, enabling before/after score comparison without manual bookkeeping.

Eval Story · Arize Phoenix

Offline + Online · Arize Phoenix

  • 4 golden BRDs (Arbor Risk, a fictional fraud-detection platform) with ground-truth expectations — rule-based structural checks run instantly, LLM-as-judge rubrics run per agent
  • Per-agent rubrics: brd_coverage, phase_coherence, classification_accuracy, existing_stack_preference, feedback_specificity
  • OpenInference AGENT spans stream live traces to Phoenix UI — per-agent input/output visible in production, not just end-to-end
  • Prompt change workflow: edit registry → bump version → re-run evals → compare side-by-side. A CLI diff script means no Phoenix UI required for the score delta.

Per-agent evaluation — not end-to-end vibe checks — is the only way to know which agent is responsible when output degrades. That's an architectural position as much as a technical one.

What this demonstrates

Multi-agent systems fail silently in ways monolithic LLM calls don't. That's how I think about building AI teams and AI systems: instrument first, then improve.

02
Applied AI at Work Vertex AI Production
Document Intelligence Pipeline — Mechanical Drawings Extraction
Onboarding coordinators were manually reading mechanical drawings and transcribing building specifications into data systems — a slow, error-prone bottleneck that scaled with every new facility.
Vertex AI Document AI Fine-Tuning Structured Extraction GCP
Details +

Architecture & Key Decisions

  • Fine-tuned on proprietary document samples Mechanical drawings have non-standard layouts — dimension annotations, title blocks, revision clouds — that off-the-shelf document AI models aren't trained to handle. Fine-tuning on internal samples was the right call over prompt engineering an inadequate base model.
  • Vertex AI for data residency and fine-tune control Proprietary building documents couldn't leave the enterprise boundary. Vertex AI Document AI provided native fine-tuning without requiring document samples to be sent to third-party training infrastructure.
  • Structured output mapped to existing data schema Extracted fields (equipment specs, dimensions, system types, building codes) were normalized to match the existing onboarding data model — eliminating re-mapping and making the pipeline a drop-in replacement for the manual step, not a parallel system to reconcile.
  • Human-in-the-loop on low-confidence extractions Below a confidence threshold, extractions were flagged for coordinator review rather than auto-populated. This preserved accuracy while still eliminating the majority of manual work — and gave coordinators a correction signal that fed back into fine-tuning iterations.

Eval Story · Extraction Quality

Document Extraction · Precision / Recall

  • Field-level precision and recall measured against a held-out set of manually verified drawings — not a synthetic test set created after training
  • Confidence threshold calibrated against coordinator correction rate — the threshold that minimized total coordinator time, not the one that maximized raw accuracy
  • Correction feedback loop: coordinator edits on flagged extractions fed into subsequent fine-tuning rounds
  • Outcome: measurable reduction in time for onboarding coordinators filling building data — removing the manual transcription step from the critical path

The right metric here wasn't model accuracy — it was coordinator time saved. That distinction shaped every threshold and architecture decision in the pipeline.

Production RAG · Embeddings Fine-Tuning IR Metrics
ShopTalk — Natural Language Product Search
Keyword search fails on conversational product queries. The real question behind every e-commerce search team: how far do off-the-shelf embeddings get you, and when does fine-tuning actually pay?
RAG Embeddings Fine-Tuning Recall@10 · NDCG@10 Cross-Encoder Reranking GCP Cloud Run
Details +

Architecture & Key Decisions

  • Three-model comparison before fine-tuning Evaluated multiple pre-trained embedding models on Recall@10 and NDCG@10 before fine-tuning any of them. e5-base won. The model selection decision was measurement-driven — not assumption-driven.
  • Triplet loss fine-tuning for domain specificity Product search queries have different semantic structure than general text. Triplet loss training on (query, positive product, negative product) triplets closed the gap that off-the-shelf embeddings leave for commerce-specific language. Published as mananthakris/e5-base-ft-abo on Hugging Face.
  • Cross-encoder reranking as a deliberate second pass Bi-encoder retrieval is fast but imprecise at the top of the ranking. Cross-encoder reranking over the top-K candidates applies the compute budget where ranking quality matters most — query-by-query relevance judgment, not bulk retrieval.
  • Production-grade deployment on GCP Cloud Run Containerized API + UI, GitHub Actions CI/CD, ChromaDB locally and Weaviate Cloud in production. VectorDB abstraction layer routes transparently between environments. 145K+ products indexed.

Eval Story · IR Retrieval Metrics

Retrieval Quality · Recall@10 · NDCG@10

  • Recall@10 used to gate model selection before any fine-tuning compute — established which base model was worth improving
  • NDCG@10 measured ranking quality, penalizing relevant results that appear lower in the list — not just whether they appear at all
  • Fine-tuning gains validated against baseline e5-base on a held-out query set not used during training
  • Published fine-tuned model: mananthakris/e5-base-ft-abo — reproducible, inspectable, reusable

Retrieval eval came first, drove model choice, then validated fine-tuning gains. Measurement → decision → measurement — not the other way around.

04
Applied AI at Work LangSmith Production
Enterprise Chatbot — Evolving a Production System with RAG and Evals
An internal enterprise chatbot with tool-calling was falling short on knowledge-heavy queries. The fix wasn't a new model. It was the right retrieval layer — and the discipline to know when it was actually working.
RAG Tool Calling LangSmith Answer Faithfulness Retrieval Evals
Details +

Architecture & Key Decisions

  • RAG added as a first-class tool, not a replacement Rather than replacing the tool-calling layer, RAG was integrated as one more tool in the agent's decision space — letting the LLM route queries dynamically between live-action tools and retrieval-backed knowledge, depending on what the query actually needed.
  • Retrieval pipeline calibrated against real query distributions Chunking strategy, embedding model, and re-ranking were chosen and iterated against actual user query patterns — not synthetic benchmarks. Enterprise queries are domain-specific; general benchmarks don't predict field performance.
  • Eval infrastructure before prompt optimization LangSmith tracing was wired in before any prompt tuning began. Every retrieval call, tool invocation, and LLM output was captured — making improvement measurable rather than intuitive. Prompt changes without a corresponding eval run were treated as untested deploys.

Eval Story · LangSmith

Production Evals · LangSmith

  • Retrieval quality tracked per query type: tool-answered, RAG-answered, and hybrid — understanding which path the LLM was choosing and whether it was choosing correctly
  • Answer faithfulness scored against retrieved context, not model confidence — a model can be confidently wrong
  • End-to-end trace logging connected prompt changes to outcome deltas, making each iteration's effect visible
  • Dataset of real user queries used to regression-test each retrieval iteration before deploying to production

The policy change that mattered most: prompt changes without a corresponding eval run were treated as untested deploys. That norm came before any of the tooling did.
